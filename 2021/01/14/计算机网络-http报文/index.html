计算机网络-http报文

计算机网络-http报文

1、http报文概念

http报文就是http通信的基本单位,由8位字节流组成

2、http报文的整体结构

整体结构:报文首部 + 空行 + 报文实体
对于请求报文和响应报文又有区别

请求报文

报文首部: 请求方法 + URI + HTTP版本 + HTTP首部字段
空行: CR + LF
报文实体: ….

响应报文

报文首部: HTTP版本 + 状态码 + HTTP首部字段
空行: CR + LF
报文实体: ….

3、HTTP报文具体组成

报文首部

报文首部主要可以区分为两块 起始行 和 http首部字段

起始行

1、请求报文的起始行:请求行
方法名 + URI + HTTP版本
例如:GET /index HTTP1.1

2、响应报文的起始行:状态行
HTTP版本 + 状态码 + 原因短语
例如 HTTP1.1 200 OK

http首部字段

这一部分在下文详细列举

空行

CR 或 LF (回车或换行符)
在报文中,空行是头部和实体的分界线,主要作用就是用于区分头部和实体,如果在头部字段中故意额外添加一个空行的话,那么接下来所有的内容全部都会被当作实体来看待

报文实体

请求体/响应体:一般请求报文的实体部分不放东西,而响应实体则会放一些响应的结果

4、首部字段

4.1 首部字段组成方式

  • http首部字段的组成通常是key:value的形式的键值对
    如:Content-type:text/html
  • 但是也有可能一个key值会有多这个value值-> key:value1,value2
    多个选项的value之间会使用”,”隔开
    如:Keep-Alive:timeout=15,max=100
  • 还有可能一个key会有可选参数 -> key- value1,q=1;value2,q=0.8
    如:Accept:text/html,q=1;application/xml,q=0.8 (这里有两个可选值,这里的q的意思是权重,优先使用哪个,范围是0-1,值越大权重越大)

4.2 首部字段的类型

基本可以分为4类

  • 1.通用首部字段:请求报文和响应报文中都会有的字段类型
  • 2.请求首部字段:请求报文的字段类型
  • 3.响应首部字段:响应报文的字段类型
  • 4.实体首部字段:报文实体部分的字段类型

这里需要记住每种类型中常用的字段

1.通用首部字段

  • 1.Cache-Control
    //TODO

  • 2.Date
    用于表示创建报文的时间和日期
    例:Date:Sun,05 Jar 2021 19:00:00 GMT

  • 3.Connection
    作用有两个:
    1、控制不再转发代理的首部字段:Connection: …(一些不会转发代理的字段名称)
    2、管理持久连接: Connection:Keep-Alive (在http/1.1 之前默认是close,在http/1.1之后默认就是Keep-Alive)

    这里还有一个首部字段的key为Keep-Alive
    可以用来设置最大的请求数和超时时间
    Keep-Alive:timeout=5,max=1000 (timeout代表一个空闲连接需要保持打开状态的最小时长——单位秒,max代表在此连接可以发送请求的最大值)
    这个字段设置的值的前提是需要设置Connection:Keep-Alive,不然设置的没有意义

  • 4.Transfer-Encoding
    作用: 规定传输的报文中主体部分使用的编码方式
    例: Transfer-Encoding:chunked(代表分块出传输)

2.请求首部字段

  • 1.Host
    作用:可以明确指出主机名或域名
    例:如果一台服务器上托管了多个域名,那么在进行DNS域名解析时,会得到相同的IP地址,因此需要在Http请求时,在Host首部进行指出完整的主机名或域名的URI

  • 2.Accept
    作用:客户端或者代理能够处理的媒体类型
    下文将详细介绍

  • 3.Authorization
    作用:web的一些认证相关的信息

  • 4.User-Agent
    作用:将创建请求的浏览器和用户代理名称等信息传给服务器
    例:user-agent:Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/87.0.4280.141 Safari/537.36

  • 5.Range
    作用:获取资源的某个范围的部分,用于断点续传
    例:Range:bytes=5001-10000
    响应报文状态行返回206状态码和Partial Content 说明 和响应头中带上Content-Range
    例:Http/1.1 206 Partial Content

     Content-Range:bytes 5001-10000
 如果无法处理则返回200 OK 和整个资源

3.响应首部字段

  • 1.Location
    作用:用于指向重定向之后的url
    一般状态码返回3xx 时,响应报文中首部字段会有Location来指示重定向的url

  • 2.Server
    作用:告知客户端当前服务器上安装的HTTP服务器应用程序的信息
    例:Server:nginx

  • 3.Age
    作用:告诉客户端,服务器在多久之前创建的了响应
    如果创建该响应的是缓存服务器,则表示缓存后的响应再次发起认证到认证完成的时间值
    例:Age:600(单位是秒)

4.实体首部字段

  • 1.Allow
    作用:告诉客户端服务器能够支持的Http方法
    例:Allow:GET,POST,HEAD
    当服务器收到不能够支持的方法时,会返回状态405 Method Not Allowed,并且会把所有支持的方法写入响应报文首部字段Allow中返回给客户端

  • 2.Last Modified
    作用:资源最后的修改时间,一般配合If-Modified-Since 失效协商缓存
    例:Last-Modified:Wed,Nov 11 2020 08:00:00 GMT

  • 3.Expires
    作用:强缓存的一种方式,设置资源缓存的失效时间
    例:Expires:Wed,Nov 11 2020 08:00:00 GMT

5.非标准首部字段

非标准首部的字段是自行扩展的

  • 1.Cookie
    作用:客户端传递给服务器的一些Cookie信息
    例:Cookie:name=xxgw

  • 2.Set-Cookie
    作用:服务器在响应报文中设置要传递给客户端的Cookie信息
    例:Set-Cookie:name=xxgw;HttpOnly

6.Accept相关字段

按照类别可以分为5类

  • 1.数据格式
    表示接收方期望收到的数据格式
    请求报文:接收方->Accept
    响应报文:发送方->Content-Type
    例:
    text:text/html,text/plain,text/css
    image:image/gif,image/jpeg,image/png
    application:application/json,application/javascript

  • 2.压缩方式
    表示采用哪种方式进行压缩
    请求报文:接收方->Accept-Encoding
    响应报文:发送方->Content-Encoding
    例:字段值:
    gzip(使用最多的)
    deflate(另外一种著名的压缩格式)
    br(一种专门为http发明的压缩算法)

  • 3.支持语言
    指定语言集优先级
    请求报文:接收方->Accept-Language
    响应报文:发送方->Content-Language

例:zh-CN,zh,en

  • 4.范围请求
    表示获取资源的某范围的部分
    请求报文:接收方->If-Range和Range
    响应报文:发送方->Accept-Ranges和Content-Range
    例:
    请求
    If-Range:”usage-11111”
    Range:bytes=5001-10000
    响应
    Http/1.1 206 Partial Content
    Accept-Range:bytes
    Content-Range:bytes 5001-10000/10000

  • 5.字符集
    指定字符的类型集
    请求报文:接收方->Accept-Chartset
    响应报文:发送方->Content-Type
    例:chartset=utf8

5.编码提升传输效率

对于通过编码来提升传输效率一般可以分为两个部分进行讲解:

  • 内容编码
  • 传输编码
    区别:

内容编码:通常对于实体内容进行压缩编码,目的就是优化传输,例如使用gzip压缩文本文件,能大幅度减小体积。且通常是有选择性的,例如jpg、png这种图片类格式的文件就不会进行再压缩,因为这类资源已经不能够再进行压缩了。
传输编码:这个指的就是来改变报文格式的,通常在头部加入了Transfer-Encoding:chunked,实现分块编码.
以上两个的相同点就是都是进行在实体主体部分的可逆变换。相当于,从一个地方将电脑主机给拆成零件,然后进行运输到目的地后,再将这些零件给拼装起来.

5.1内容编码

内容编码介绍

表示http应用程序在发送请求之前会对内容进行编码,使用Content-Encoding来指定编码格式,Content-Length来指定文件的大小
具体过程:

  • 1.服务器生成原始的响应报文,有Content-Type和Content-Length首部字段
  • 2.将原始的响应报文经过内容编码服务器创建编码后的报文。这个编码后的报文同样有Content-type和Content-Length两个字段。但是其中Content-Length字段会改变,会变小,因为进行编码后文件肯定会变小。
  • 3.然后会新增一个Content-Encoding 首部字段,目的是执行编码的方式,这样收到的应用程序就知道需要使用哪种格式去进行解码。
  • 4.接收程序收到响应后,根据Content-Encoding去进行解码得到原始的响应报文

内容编码类型

  • Accept-Encoding(请求报文)
    是在请求报文定义的首部字段,用于告诉服务器客户端能够支持哪种编码方式,服务器会根据这些编码方式来进行选择其中一项进行编码,如果没有定义Accept-Encoding则服务器会默认认为是能够接受任何方式的编码即Accept-Encoding:*

  • Content-Encoding(响应报文)
    是在响应报文定义的首部字段,作用就是告诉客户端采用了哪种编码方式,客户端会根据该编码方式进行解析

内容编码代号

gzip:实体采用GUN zip 编码
compress:实体采用Unix的文件压缩程序
deflate：实体使用zlib的格式压缩
identity：不对实体进行编码。当没有Content-Encoding首部的时候，就默认这种情况

内容编码与HTTP压缩

Http压缩,在http协议中,其实就是内容编码的一种

内容编码的方式有很多种,可以使用如gzip的方式将内容进行压缩,从而可以减少实体的传输时间

也可以使用其他的编码对内容进行加密处理,以防三方进行拦截数据

http压缩的具体过程

Http压缩就是服务器与客户端之间压缩传输的文本内容的方法

http采用通用的压缩算法gzip,对html、js、css文件进行压缩

例:Accept-Encoding:gzip

  • 1.浏览器发送请求,并且请求首部字段中有Accept-Encoding:gzip
  • 2.服务器接受到请求后,生成原始的响应报文,其中有Content-Type:… 和 Content-Length:1000 这两个字段
  • 3.服务器在对原始报文实体部分进行gzip编码处理。编码后在首部字段中将Content-Length的值改为编码处理后的长度,并且新增Content-Encoding:gzip字段,再将编码后的报文发送给客户端
  • 4.客户端收到请求后,根据响应首部字段中的Content-Type的编码方式对实体进行解码,得到原始的响应报文

gizp压缩的本质就是将相同的字符替换,从而得到减小文件的体积

5.2传输编码

传输编码就是对内容编码之后的tranfer-Encoding再进行编码,对编码后的内容进行分块,每一块都有长度标识
